You are an expert full-stack developer. Create a complete web-based control panel for a hospital patient alarm system. The system will later run on an ESP32‑S3 master device, but for now we are building and testing it in Replit using Node.js and Express. The final code must be clean, modular, and easily portable to an ESP32 with the ESPAsyncWebServer library.

## Project Overview
- **Master:** ESP32‑S3 hosting a Wi‑Fi hotspot and a web server.
- **Slaves:** IoT devices (one per patient bed) that connect to the master’s Wi‑Fi. Each slave has a physical button. When pressed, the slave sends an HTTP request to the master, and the master’s website displays an alert.
- **Admin:** A separate password‑protected page to add/edit patient information for each slave.

## Functional Requirements

### 1. Main Dashboard (Public)
- Display a grid/list of all registered slaves.
- For each slave, show:
  - Patient name (editable via admin)
  - Bed number / room (editable)
  - A large button (or indicator) that lights up when the patient presses the call button.
  - The last alert time.
- When a slave sends an alert:
  - The corresponding button/indicator on the dashboard changes color (e.g., red) and shows "ALERT".
  - A visible notification (e.g., a toast or banner) appears on the dashboard for a few seconds.
  - The alert can only be cleared by an admin action (see below).
- Alerts must be **one‑shot**: pressing the button again while an alert is active does nothing until the alert is cleared.

### 2. Admin Page (Protected)
- URL: `/admin`
- Login: username `admin`, password `admin1234` (hard‑coded for now).
- After login, show a table of all slaves with the following editable fields:
  - Slave ID (unique identifier, e.g., MAC address or custom ID)
  - Patient Name
  - Bed Number
  - Room
  - (Optional) Any other basic info.
- Buttons:
  - **Add Slave** → Opens a form to enter a new slave ID and patient data.
  - **Edit** → Modify existing slave data.
  - **Delete** → Remove a slave from the system.
  - **Clear Alert** → A button next to each slave to manually clear an active alert (this should reset the dashboard indicator for that slave).

### 3. REST API Endpoints (for slaves and admin)
All endpoints return JSON.

- `POST /api/register`  
  Called by a slave when it first connects.  
  Body: `{ "slaveId": "..." }`  
  Response: `{ "success": true, "message": "Registered" }` or error if ID unknown.  
  (Slaves are added via admin first, so unknown IDs are rejected.)

- `POST /api/alert`  
  Called by a slave when its button is pressed.  
  Body: `{ "slaveId": "..." }`  
  Response: `{ "success": true, "alertId": "..." }`  
  If an alert for that slave is already active, return `{ "success": false, "reason": "alert already active" }`.

- `GET /api/slaves`  
  Returns a list of all slaves with their current data and alert status. Used by the dashboard to refresh.

- `PUT /api/slaves/:slaveId` (admin only)  
  Update patient data for a slave. Body includes fields to update.

- `DELETE /api/slaves/:slaveId` (admin only)  
  Remove a slave.

- `POST /api/slaves` (admin only)  
  Add a new slave. Body: `{ "slaveId": "...", "patientName": "...", "bed": "...", "room": "..." }`

- `POST /api/clearAlert/:slaveId` (admin only)  
  Clear the active alert for a specific slave.
